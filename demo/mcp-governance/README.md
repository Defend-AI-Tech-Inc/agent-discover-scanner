# Day 1 MCP Governance — Three Commands

This demo shows how to go from zero to enforced MCP policy in three commands,
using AgentDiscover Scanner to discover what MCP servers exist and mcpfw to
enforce governance on them.

---

## Prerequisites

```bash
pip install agent-discover-scanner mcpfw-defendai
```

---

## Three-command workflow

### 1. Discover what MCP servers exist in your project

```bash
agent-discover export-mcpfw-policy . --stance balanced --output mcpfw-policy.yaml
```

The scanner reads all MCP config files on this machine (Claude Desktop, Cursor,
Windsurf, VS Code, Gemini CLI, project-level `.mcp.json`, etc.) and generates
a ready-to-enforce mcpfw policy YAML.

Stances:
- `strict` — deny-by-default, minimal tool allowlists, full DLP blocking
- `balanced` — allow with DLP and allowlists for unverified servers (default)
- `monitor` — log only, no blocking — use for Day 1 observability

Output: `mcpfw-policy.yaml` listing discovered servers and defensive rules.

### 2. Review the generated policy

```bash
cat mcpfw-policy.yaml
```

The file contains:
- Global DLP rules (credential detection, prompt-injection detection) applied to
  all MCP server responses
- Per-server rules derived from what the scanner found: filesystem path
  restrictions, blocked tool lists, rate limits, unverified-server allowlists

Edit the file if you need to adjust paths, add servers, or change the stance
for a specific server.

### 3. Enforce the policy on any MCP server

```bash
mcpfw wrap --policy mcpfw-policy.yaml -- npx @modelcontextprotocol/server-filesystem /tmp
```

`mcpfw wrap` sits between your AI client and the MCP server. Every tool call
and tool response passes through the policy engine before being forwarded.
Blocked calls produce an `isError: true` response; all decisions are written
to a JSONL audit log.

## What the scanner detects

| Source | What it reads |
|---|---|
| Claude Desktop | `~/Library/Application Support/Claude/claude_desktop_config.json` |
| Cursor | `~/.cursor/mcp.json` |
| VS Code | `.vscode/mcp.json` |
| Gemini CLI | `~/.gemini/settings.json` |
| OpenAI Codex | `~/.codex/config.toml` |
| Project-level | `mcp.json`, `.mcp/config.json`, `.mcp.json` |
| package.json | Dependencies containing `mcp` in the name |
| requirements.txt | Python packages containing `mcp` in the name |
| Network traffic | Live connections to known MCP endpoints (Stripe, Salesforce, etc.) |

---

## Running inside scan-all (continuous mode)

To emit a policy file as part of the full 4-layer scan:

```bash
agent-discover scan-all . --emit-mcpfw-policy mcpfw-policy.yaml --stance strict
```

This runs Layers 1–4, then writes the policy based on all MCP findings
(including network-detected servers from Layer 2).

## Sample policy

`sample-policy.yaml` in this directory was generated by scanning the
agent-discover-scanner repo itself:

```bash
agent-discover export-mcpfw-policy /path/to/agent-discover-scanner \
    --stance balanced --output sample-policy.yaml
```

It found 9 MCP servers (from Claude Desktop config and project-level configs)
and generated 7 global DLP rules plus per-server rules for filesystem servers.

## Attack coverage

The generated policy defends against the attack classes documented in
[mcpfw's attack scenario library](https://github.com/Defend-AI-Tech-Inc/mcpfw):

| Attack | Policy rules generated |
|---|---|
| Shadow Server | `blocked_tools`, `allow_paths`, response DLP |
| Rug Pull | `allowed_tools` allowlist, `verify_server_tools` (strict) |
| Shadow Exfil | `block_patterns` on args, credential DLP |
| Meter Is Running | `detect-retry-injection`, `detect-pagination-injection` |
| Overprivileged Agent | `allowed_tools` + `allow_paths` |
